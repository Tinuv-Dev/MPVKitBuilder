# 提前转录 · 字幕策略研究（分段 / 显示 / 逐字高亮）

> 适用前提：PoC 已验证两路径（Path A 内解码 PCM、Path D 解压缩包）均能拿到**超前于播放头**的 PCM，并喂 `SpeechTranscriber` 出文本（见 `提前转录-spec-tasks.json` 与 `IovisPCMTapPatch.md`）。
> 本文只解决「拿到文本之后，字幕怎么分段、怎么正确显示、怎么逐字高亮」的策略问题，不涉及补丁/解码/链路本身。
> 现状参考来自 IovisKit 仓库 `git stash@{0}` 的实验原型（原型级、不作工程范本）。
> API 事实已用 Apple 官方 Speech 框架文档核实，标注于文末。

---

## 0. 最重要的前提：这是「提前转录」，不是「实时转录」

`SpeechTranscriber` 的默认设计面向**实时贴播放头**的场景（`progressiveTranscription` preset + `volatileResults`），它给两类结果：

- **volatile（临时）**：边听边改、会被后续结果覆盖。用来做「话音未落先上字」的低延迟体验，代价是**抖动 / 文本回跳**。
- **final（最终）**：模型确信后定稿，`range`（`CMTimeRange`）稳定不再变。

本项目架构是**在播放头前方 N 秒解码 + 转录**（lookahead）。由此推出一个核心红利：

> 当画面播到某句话时，那句话的 **final 结果早已转完并落库**。因此常态下**根本不需要 volatile**。

这把「实时转录」的两难（要低延迟就得忍抖动）直接消解：提前转录可以**只用 final**，可以**从容做高质量分段**，可以在显示前就把整句的逐词时间戳备好。

> 原型偏差：`IovisPCMTranscriptionSink` 同时 `setLatest`（含 volatile 兜底）+ `appendFinal`，是把实时方案照搬过来。提前转录下 volatile 兜底反而引入抖动，应砍掉。

**策略结论 0**
- `reportingOptions` 常态**去掉 `.volatileResults`**，只消费 final 落库。
- 保留 `attributeOptions = [.audioTimeRange]`（逐字高亮的数据来源），可选加 `.transcriptionConfidence`。
- 仅当**领先量不足**（seek 清空 lookahead、起播瞬间还没转出）时，降级为贴播放头实时模式，那时才临时启用 volatile 兜底。把它做成**显式状态切换**，而非无条件兜底。

## 1. 如何分段（Segmentation）

字幕分段是**两层**，必须分开看。

### 1.1 第一层 · 识别分段（模型决定，可干预）

- **被动**：模型在自然停顿 / 语义点处 finalize，你接收到一条条 final 结果，每条带 `result.range`（`CMTimeRange`）。一条可能是一整句长句（十几秒、几十个词）。
- **主动干预**：`SpeechAnalyzer.finalize(through: CMTime)` 可**强制在指定时间码切一刀定稿**。提前转录场景下时间充裕，可在静音间隙（VAD，可选配 `SpeechDetector`）主动调用，得到更整齐的句段边界。
- ⚠️ **事实更正**：`SpeechTranscriber.ReportingOption` **没有** `frequentFinalization`（那是 `DictationTranscriber` 的）。想要更碎的定稿只能靠 `finalize(through:)` 自己驱动。

### 1.2 第二层 · 显示分段（字幕规范，必须自己做）

把一条（可能很长的）final 结果切成「可上屏的 cue」。按优先级套以下约束：

1. **时长**：单条 cue 建议 1.5～7s，理想 ≤ 5s；过短合并相邻、过长拆分。
2. **行 / 字数**：每行 ≈ ≤ 37 西文字符 / ≤ 16 全角 CJK 字符，最多 2 行。
3. **断点选择**：标点 > 词间静音间隙（用相邻词 `audioTimeRange` 的 gap，> ~0.4s 优先断）> 语法弱连接点。**永远在词边界断，绝不切词中间**——逐词 `audioTimeRange` 已提供精确边界。
4. **阅读速度**：单条 cue 的 `字符数 / 时长` 不超过阅读速率上限（西文 ~17 cps、中文 ~9 cps）；超了就延长 cue 时间窗或继续拆。

> 原型偏差：把「1 条 final = 1 条 cue」直接显示，缺第二层分段器。`IovisTranscriptCue { start, end, words }` 结构本身正确，缺的是「长 final → 多条规范 cue」的切分逻辑。

## 2. 如何正确显示（Display）

### 2.1 时间对齐（最易出 bug）

- 原型用全局 `baselinePts`（第一帧喂入的 media pts）+ 结果相对时间还原 media 时间。单调连续播放下能用。
- **失效场景**：seek / 换音轨 / 循环播放会让 baseline 失效，cue 时间整体错位。
- **正解**：维护显式的 **`分析器时间轴 → media 时间轴` 映射**，在每次 seek / track 切换时**重锚**（与 spec t1 「seek 与 range 合并后须重锚游标」对应，显示层须同步重锚时间映射），而非依赖单个全局 `baselinePts`。
- 注意原型记录的坑：直接把领先 ~30s 的 pts 当 `AnalyzerInput(bufferStartTime:)` 喂，分析器**不出结果**。所以喂入时间从 0 起算、用映射加回 media 时间——这条经验保留，但映射要可重锚。

### 2.2 显示驱动

- `CADisplayLink` 取当前播放秒 → `store.activeCue(at:)` → 渲染。原型这套正确。
- `activeCue` 用 `start <= time <= end + linger` 选活动 cue 合理；但**分段器须保证 cue 之间无重叠**，否则同一时刻命中两条。
- `lingerSeconds`（尾部停留）保留以免末尾瞬间消失；但下一条 cue 到来时必须让位，不能让 linger 压住新 cue。

### 2.3 volatile 取舍（承接第 0 节）

- 常态：**不显示 volatile**，只显示已落库 final cue，稳定无抖动。
- 降级：领先量不足时切到贴播放头实时模式，临时启用 volatile 兜底。做成明确状态切换。

### 2.4 置信度（可选增强）

`attributeOptions` 加 `.transcriptionConfidence` 后，低置信度词可弱化显示或不参与高亮，避免把明显识别错的词高亮给用户。

## 3. 如何逐字高亮（Word-level Highlight）

### 3.1 原型的错误

`IovisPlayertvOSSubtitleOverlayView.highlightedCharCount` 按**整句时间窗线性铺字符数**：

```
frac = (time - cue.start) / (cue.end - cue.start)
highlightChars = total * frac        // 整句匀速，错误
```

它**完全没用**已采集好的逐词 `audioTimeRange`（`IovisTranscriptWord.start/end`）。后果：语速不均时高亮与实际念到的字严重错位——念得快的词高亮跟不上，停顿处高亮还在匀速爬。原型注释「不依赖逐词时间」等于主动放弃了 API 最有价值的数据。

### 3.2 正确做法：逐词 `audioTimeRange` 驱动

每个词的 `start/end` 来自 `result.text` 的 per-run `.audioTimeRange`（`IovisPCMTranscriptionSink.makeCue` 已正确提取）。高亮逻辑：

**词级（够用、稳）**
- `t >= word.end`：该词「已念完」，全亮。
- `word.start <= t < word.end`：该词「正在念」，为当前词。
- `t < word.start`：未亮。

**字级卡拉OK（更顺滑，可选）**
仅在「正在念」的当前词**内部**做线性插值，把插值范围从「整句」收窄到「单词」，误差被限制在一个词内：

```
for word in cue.words {
    if t >= word.end            { word 全亮 }
    else if t >= word.start {
        let frac = (t - word.start) / (word.end - word.start)
        当前词高亮前 frac 比例的字符      // 字级插值仅限本词内
    } else                      { word 不亮 }
}
```

### 3.3 CJK / 字素计数

- 中文转录的 run 粒度可能是词或字、且无空格。若 run 为词级，用「当前词内字级插值」即可。
- **字素计数统一走 `String` 字素簇 / `String.Index`**，不要用 UTF-16 长度。原型用 `NSString.length` + `NSRange`（UTF-16），CJK 基本无碍但 emoji / 组合字会切错。

## 4. 策略总表

| 维度 | 原型现状 | 提前转录应采取的策略 |
|---|---|---|
| 结果类型 | volatile + final 混用 | 常态只用 **final**；领先不足降级才用 volatile |
| 识别分段 | 被动接收 final | 可选 `finalize(through:)` 在静音间隙主动切 |
| 显示分段 | 1 final = 1 cue | 加**显示分段器**：时长 / 行宽 / 词边界 / 阅读速率 |
| 时间对齐 | 全局 `baselinePts` | 显式**分析器↔media 时间映射**，seek/track 重锚 |
| 逐字高亮 | ❌ 整句线性铺字符 | ✅ **逐词 `audioTimeRange` 驱动** + 当前词内字级插值 |
| 字素计数 | NSString UTF-16 | `String` 字素簇（CJK / emoji 安全） |
| 置信度 | 未用 | 可选 `.transcriptionConfidence` 弱化低置信词 |
| 存储 | `IovisTranscriptStore` 锁 + 排序 | 结构 OK；改为只存规范化后的显示 cue |

## 5. 工程缺口（未拆 task，仅记录）

从原型到工程级，字幕策略层最大的两个缺口：

1. **显示分段器**：把长 final 结果按第 1.2 节四约束切成规范 cue（词边界对齐）。
2. **逐词高亮重写**：用 `audioTimeRange` 驱动替换整句线性插值（第 3.2 节）。

附带项：时间映射重锚（2.1）、volatile 降级状态机（2.3）、字素簇计数（3.3）。

> 按用户要求，本文不改动 `提前转录-spec-tasks.json`；上述缺口待后续工程阶段再决定是否拆为 task。

## 附录 · API 事实核实（Apple Speech 框架文档）

- `SpeechTranscriber.ReportingOption` = `{ volatileResults, fastResults, alternativeTranscriptions }`——**无** `frequentFinalization`（后者属 `DictationTranscriber`）。
- `SpeechTranscriber.ResultAttributeOption` = `{ audioTimeRange → AttributeScopes.SpeechAttributes.TimeRangeAttribute, transcriptionConfidence → ConfidenceAttribute }`。
- `SpeechModuleResult.range: CMTimeRange`——结果适用的音频时间范围。
- `SpeechAnalyzer.finalize(through: CMTime?)`——强制在时间码定稿；`cancelAnalysis(before: CMTime)`——丢弃旧音频。
- `SpeechAnalyzer.bestAvailableAudioFormat(compatibleWith:)`——**不做重采样**，以保持 `CMTime` 采样精确。
- 相关 preset：`progressiveTranscription` / `timeIndexedProgressiveTranscription`（实时）、`transcription` / `timeIndexedTranscriptionWithAlternatives`（离线/带时间索引）。
- 可用性：`SpeechAnalyzer` / `SpeechTranscriber` 为 tvOS 26 / iOS 26 / macOS 26 / visionOS 26+。
